# NestJS Vault Multi-Service Local Demo

Local Docker Compose environment for testing production-style HashiCorp Vault Dynamic PostgreSQL Secrets, MongoDB 4.0/7.0 dynamic secrets, Redis KV secrets, and RabbitMQ KV secrets with five NestJS microservices.

## Architecture

```text
                         Vault dev server
                              |
          database/creds/<per-service-role> dynamic leases
                              |
PostgreSQL appdb <----- product/customer/inventory/order/payment services
                              |
        Redis KV secret and RabbitMQ KV secret remain available
                              |
       MongoDB 4.0/7.0 dynamic leases for product/order services

order-service -> product-service
order-service -> customer-service
order-service -> inventory-service
payment-service -> order-service
```

Shared code lives in `services/shared`:

- `VaultClient`
- `VaultAuthProvider`
- `DynamicDbCredentialProvider`
- `DatabaseConnectionManager`
- `MongoDynamicCredentialProvider`
- `MongoLeaseManager`
- `MongoConnectionManager`
- health and metrics controllers
- HTTP client

Each service owns only its domain source folder under `services/<service-name>`.

## Services

| Service | Port | Vault DB role | Tables |
| --- | ---: | --- | --- |
| product-service | 3001 | `product-service-role` | `products` |
| customer-service | 3002 | `customer-service-role` | `customers` |
| inventory-service | 3003 | `inventory-service-role` | `inventory` |
| order-service | 3004 | `order-service-role` | `orders`, `order_items` |
| payment-service | 3005 | `payment-service-role` | `payments` |

Each service receives its own Vault lease and its own PostgreSQL username/password generated by Vault.

MongoDB dynamic secrets are enabled only for services that need Mongo:

| Service | Mongo version | Vault Mongo role | Mongo collection |
| --- | --- | --- | --- |
| product-service | 4.0 | `mongo40-product-role` | `product_audit_logs` |
| order-service | 7.0 | `mongo70-order-role` | `order_events` |

Vault also bootstraps `mongo40-order-role` and `mongo70-product-role` so either service can be switched between MongoDB 4.0 and 7.0 by changing `MONGO_HOST`, `MONGO_VERSION`, `MONGO_VAULT_DB_CONFIG`, and `MONGO_VAULT_DB_ROLE`.

## Start

```bash
docker compose down -v
docker compose up -d vault postgres redis rabbitmq mongo40 mongo70
./scripts/init-vault.sh
docker compose up -d --build product-service customer-service inventory-service order-service payment-service
```

If port `3001` is occupied by local AnythingLLM, stop it first:

```bash
docker stop anythingllm
```

```bash
curl http://127.0.0.1:3004/health/ready

```

Each service also exposes:

- `/health/live`
- `/health/startup`
- `/metrics`

Readiness includes Vault, PostgreSQL lease status, PostgreSQL, Mongo status, Redis, and RabbitMQ status. Mongo health reports status, version, lease state, renewal timestamps, and remaining TTL only; it does not expose usernames or passwords.

## Test Flow

```bash
npm run build
npm test -- --runInBand
./scripts/test-microservice-flow.sh
./scripts/test-mongo-vault.sh
```

The flow script validates:

- health endpoints for all services
- product/customer/inventory CRUD path
- order creation with product/customer/inventory validation
- payment creation with order validation
- Vault dynamic credential generation per role
- distinct generated DB usernames
- metrics endpoint presence
- manual revoke of one product-service lease
- product-service recovery while other services stay ready

The Mongo script validates:

- MongoDB 4.0 and 7.0 root connectivity
- Vault dynamic credential generation for all Mongo roles
- dynamic credential login against MongoDB 4.0 and 7.0
- product audit log writes through product-service
- order event writes through order-service
- lease renewal and revoke API calls

## Inspect Vault

Fetch a credential manually:

```bash
export VAULT_ADDR=http://127.0.0.1:8200
export VAULT_TOKEN=root
curl -sS -H "X-Vault-Token: $VAULT_TOKEN" \
  "$VAULT_ADDR/v1/database/creds/product-service-role"
```

List leases:

```bash
curl -sS -H "X-Vault-Token: $VAULT_TOKEN" \
  -X LIST "$VAULT_ADDR/v1/sys/leases/lookup/database/creds/product-service-role"
```

Revoke one lease:

```bash
curl -sS -H "X-Vault-Token: $VAULT_TOKEN" \
  -H "Content-Type: application/json" \
  -X POST \
  -d '{"lease_id":"database/creds/product-service-role/<lease-key>"}' \
  "$VAULT_ADDR/v1/sys/leases/revoke"
```

Watch recovery:

```bash
docker compose logs -f product-service
curl http://127.0.0.1:3001/health/ready
```

## Bootstrap Details

`scripts/postgres-init/001-schema.sql` creates all service tables with admin credentials.

`scripts/init-vault.sh` configures:

- KV v2 secrets for Redis/RabbitMQ/static DB location metadata
- Vault database mount
- `database/config/appdb`
- five per-service Vault database roles
- table-specific grants only
- KV v2 bootstrap metadata for `secret/data/dev/mongo40` and `secret/data/dev/mongo70`
- `database/config/mongo40` and `database/config/mongo70`
- Mongo roles `mongo40-product-role`, `mongo40-order-role`, `mongo70-product-role`, and `mongo70-order-role`
- Mongo role grants limited to `readWrite` on `appdb`
- TTL `60s`, max TTL `180s`

## Security Notes

- Services do not receive static DB username/password.
- Generated DB credentials are held in memory only.
- Generated Mongo credentials are held in memory only.
- Mongo pools rotate credentials without restarting the service.
- Mongo leases are renewed automatically and rotate on renew failure, low TTL, expiration, or startup.
- Logs include role, lease id, TTL, and rotation reason.
- Logs/API do not expose username, password, token, or authorization headers.
- Each service role can access only its own tables.
- Health endpoints and metrics do not expose secrets.

## Kubernetes Migration Notes

The architecture is ready to replace only environment/deployment concerns:

- replace local token auth with Kubernetes Auth, AppRole, OIDC/JWT, IAM, etc.
- change `VAULT_ADDR`
- deploy each service as its own Deployment
- optionally use Vault Agent, Injector, CSI Driver, or service mesh
- keep shared lease and DB connection management code unchanged
